# Takeover Guide

> 这份文件是给新 agent 的接手入口，不属于项目开发文档。

## 阅读顺序

新 agent 接手时，按这个顺序读取：

1. `AGENTS.md`
2. `memory.md`
3. `handoff.md`
4. `docs/CODE_STYLE.md`
5. `README.md`
6. `docs/README.md`
7. 对应模块文档
8. 代码
9. 测试

## 接手检查清单

- 当前任务到底是“规范化”还是“扩功能”
- 当前哪些模块属于 Active
- 当前哪些模块属于 Frozen
- 当前文档是否已经同步到代码
- 当前测试是否真实覆盖已有功能
- 当前是否存在执行环境故障

## 处理原则

- 先验证事实，再动代码。
- 先修结构和规范，再修表现层。
- 先保证主链路闭合，再考虑扩展模块。
- 不要根据旧对话脑补实现细节。

## 对新 agent 的硬约束

- 代码规范以 `docs/CODE_STYLE.md` 为唯一准绳。
- 如果行为没问题，就不要为“看起来更优雅”而重构过度。
- 如果要拆文件，必须按职责拆，不要制造新的中间层。
- 如果文档与代码不一致，修完代码后必须同步文档。

## 遇到阻塞时怎么做

- Shell 不能用：先确认环境，不要盲改。
- 测试跑不起来：先定位是环境问题还是代码问题。
- 多 agent 冲突：立即缩小改动范围，避免互相覆盖。
- 需求含糊：优先遵守 `AGENTS.md`、`memory.md` 和当前代码事实。
