# AGENTS

> 这份文件是给 EleBot 项目中的 Codex agent 用的总规则，不属于正式项目文档。

## 1. 目标

EleBot 当前目标很明确：

- 做成 nanobot 风格的终端 AI 助手
- 先完成已有核心能力
- 不扩未来能力
- 不做兼容层
- 不保留过渡实现

你在这个项目里的第一优先级，不是“设计更多东西”，而是让当前代码结构、主链路、测试和文档保持清晰、真实、可维护。

## 2. 接手顺序

新 agent 接手时，必须按下面顺序读取项目信息：

1. `AGENTS.md`
2. `memory.md`
3. `handoff.md`
4. `takeover.md`
5. `docs/CODE_STYLE.md`
6. `README.md`
7. `docs/README.md`
8. 对应模块文档
9. 代码
10. 测试

不允许跳过前四项后直接改代码。

## 3. 当前范围

### Active 模块

这些模块属于当前默认主链路：

- `cli`
- `agent`
- `providers`
- `session`
- `config`
- `command`
- `bus`
- `utils`
- `templates`
- 最小工具调用闭环

### Frozen 模块

这些模块保留，但当前不进入默认启动链路：

- `channels`
- `api`
- `cron`
- `heartbeat`
- `skills`
- `security`
- `bridge`

Frozen 的意思不是删除，而是：

- 保留目录和代码
- 当前不接入主链路
- 当前不作为核心验收范围

## 4. 绝对约束

- 不做兼容层。
- 不做迁移脚本。
- 不做临时回退逻辑。
- 不为未来能力提前铺大框架。
- 不把主逻辑堆进 `__init__.py`。
- 不在看不到代码和测试的情况下盲改实现。
- 不把交接文档写进 `docs/`。
- 不把计划文档当成模块文档。

## 5. 代码规范

代码规范唯一入口是 `docs/CODE_STYLE.md`。

如果你要改代码，必须遵守：

- 所有函数和方法都应有清晰中文 docstring。
- docstring 要说明参数和返回值。
- 变量命名必须清晰易懂。
- 禁止无意义缩写。
- 注释使用中文。
- 注释只解释“为什么”，不要解释显而易见的“做了什么”。
- 不要添加不必要的类、封装和中间层。
- 改动以最小解、最优解为准。

## 6. 文档规则

- `README.md` 只做项目介绍、快速开始和文档索引。
- `docs/` 只放正式项目文档。
- 根目录的 `memory.md`、`handoff.md`、`takeover.md` 只给 agent 交接使用。
- 文档必须反映真实代码，不允许超前描述未落地能力。
- 代码改了，对应文档必须同步。

## 7. 测试规则

- 测试只覆盖项目当前已有功能。
- 不为 Frozen 模块补未来测试。
- 不接受只有 happy path 的测试。
- 主链路变更后，必须检查对应测试是否仍然成立。
- 如果测试跑不通，要先判断是环境问题还是代码问题。

## 8. 工作方式

### 允许做的事

- 清理结构冗余
- 统一代码规范
- 修正文档和代码不同步
- 补齐已有能力缺少的测试
- 让主链路更清晰，但不改变既有目标

### 不允许做的事

- 擅自改产品方向
- 擅自恢复 Frozen 模块到默认入口
- 看到旧方案就补兼容
- 为了“更优雅”而过度重构
- 没确认边界就跨模块大改

## 9. 多 Agent 协作规则

- 每个 agent 只能改自己负责的目录。
- 多个 agent 不要同时修改同一批文件。
- 如果发现职责冲突，先收口边界，再继续动手。
- 不要回滚其他 agent 的改动，除非明确要求。
- 如果环境异常，先汇报，不要硬改。

## 10. 遇到问题时的判断顺序

遇到冲突或不确定时，按这个优先级判断：

1. 当前代码事实
2. `docs/CODE_STYLE.md`
3. `AGENTS.md`
4. `memory.md`
5. `handoff.md`
6. 模块文档
7. 历史对话

历史对话优先级最低，不能压过代码事实。

## 11. 当前阶段的正确推进顺序

1. 先确认环境可用
2. 再确认代码结构是否符合当前文档
3. 再做代码规范化
4. 再补测试和文档同步
5. 最后做主链路联调

不要跳步骤。

## 12. 一句话原则

EleBot 当前阶段的 agent 工作原则只有一句话：

**先相信代码事实，按规范做最小修改，把已有核心能力做清楚，不扩、不绕、不兼容。**
